## Найденные баги

### BUG-001 [CRITICAL] [ИСПРАВЛЕН] WorkersStep: workerId напарника = undefined
- Файл: `app/src/components/terminal/WorkersStep.tsx:64-77`
- Симптом: при производстве с напарником API возвращает 400 "Ошибка валидации"
- Причина: API terminal/auth возвращает `{id, name, role}`, а WorkersStep читал `data.workerId` вместо `data.id`
- Запрос уходил как `{"workerId": undefined, "quantity": 5}` → Zod отклонял
- Фикс: заменено `data.workerId` на `data.id` в verifyPin()
- Воспроизведение: Терминал → деталь → количество → "С напарником" → PIN напарника → количество → Добавить → Подтвердить → 400

### BUG-002 [MINOR] React key warning в WorkersStep
- Файл: `app/src/components/terminal/WorkersStep.tsx`
- Console: "Each child in a list should have a unique key"
- Появляется при добавлении напарника в список рабочих

### BUG-003 [DESIGN] PIN напарника перезаписывает cookie сессии
- POST /api/terminal/auth для проверки PIN напарника устанавливает Set-Cookie
- После проверки PIN напарника cookie меняется с Иванова на Петрова
- Все последующие запросы идут от имени Петрова, а не Иванова
- Пока не ломает функционал (produce использует workers из body, не из cookie для начисления), но может стать проблемой
- Решение: отдельный endpoint для проверки PIN без Set-Cookie, либо флаг `noCookie`

### BUG-004 [MINOR] React key collision на странице Выработка
- Файл: `app/src/app/warehouse/production/page.tsx`
- Console: "Encountered two children with the same key"
- Появляется при отображении логов с несколькими рабочими в одной операции
- Предположительно: ключ строки таблицы не уникален когда одна операция содержит двух рабочих
